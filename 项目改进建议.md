P0：拆分网关巨石文件（最高优先）
src/gateway/server.ts 已超大且承担 RPC、流控、审批、文件系统、provider 管理等多职责，是当前最大维护风险。先按领域拆成 rpc/*、agent-runtime/*、security/*、events/*，再加模块边界测试。

P0：清理“已废弃但仍存活”的 plans/ideas/worktree 双轨逻辑
同一文件里既有 deprecated 返回，又有旧实现分支，容易引入行为分叉。建议统一到 goals/tasks 主线，删除旧 RPC 分支并做一次迁移脚本与兼容窗口公告。

P0：补齐关键路径自动化测试
优先覆盖“最容易出事故”的链路：chat.send -> tool approval -> agent.resume、scheduler run -> message delivery、ws replay/reconnect。可先扩展现有脚本目录 scripts，再进 CI。

P1：统一能力声明与实际实现（避免配置误导）
src/config.ts 的 minimax、文案中的 Slack 与当前运行能力不一致。建议把“已支持/实验中/预留”分层，并在 provider.detect 和 UI 显式标注状态。

P1：加强 RPC 输入输出 schema 校验
网关 RPC 很多分支在手工处理参数，建议统一用 zod schema（入参和回参）并集中错误码，减少隐式行为和前后端字段漂移。

P1：提升任务计划审批模型的状态机严谨性
src/tools/tasks.ts 已有不错约束，但建议把状态转移做成显式状态机，禁止“非法跃迁”在网关层漏过，避免并发下状态污染。

P1：建立可观测性基线（结构化日志 + 指标）
现在日志较分散，建议统一 run_id/session_key/tool_name 字段，增加关键指标：平均响应、审批等待时长、replay 延迟、失败率，方便定位线上问题。

P2：收敛安全策略配置入口
路径白名单、工具策略、频道策略分散在多处。建议增加单一安全配置模型和冲突检测（比如 allow/deny 冲突提示），并做“配置体检”命令。

P2：桌面端状态管理降复杂度
desktop/src/hooks/useGateway.ts 过重。建议拆成 connection/session/stream/approvals 四个 hooks，降低回归风险并提升可测试性。

P2：发布与工程标准化
补上统一 CI（typecheck + 核心测试 + 打包冒烟），并固定 Node/依赖策略，减少“本地能跑、发布失败”的概率。
